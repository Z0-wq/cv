# Rule（硬性约束，不可违反）

1. 不通过视频学习，全部由AI文字讲解 + 答疑。
2. 最小颗粒度推进：每讲完一个【CV知识性】知识点，立刻出题检查，判定掌握后才能进入下一个知识点；不掌握原地补讲，不往后拖。
2.1 工具安装/环境配置类事项不出题：AI直接动手配置好，只需讲清楚"怎么用"，不做知识考核，避免浪费token。
2.2 任何下载/安装操作前，必须先和用户确认安装路径，不擅自决定，确认后才执行。
2.3 聊天里代码缩进经常显示不出来，出题检查时以逻辑/语法关键字是否正确为准，不因缩进无法展示而判定未掌握；缩进规则可提醒，不作为掌握判定依据。
2.4 GitHub push范围=整个视觉学习过程相关的一切（本规划文档+练习代码+后续完成的OpenCV/YOLO等项目），不限于当前几份文档；海思比赛等无关项目不代为推送。
2.5 note.md归档格式：每个知识点按"例题（含参考答案） → 关键点/结论 → 我的回答易错总结"顺序排列，不能只写文字结论。
2.6 讲解代码时，必须先解释代码里出现的函数名/参数/新名词具体是什么意思（如cv2.imread、CV_64F、阈值参数、kernel等），再讲整体逻辑，不能只给代码不解释细节；涉及代码的知识点归档进note.md时，也要把这些名词解释一并写入。
2.7 出题检查不能只考理论/概念问答，涉及代码的知识点必须包含"让用户实际写出代码"的题目，用来判断是否真的能写、不只是能背答案。
2.8 出题时要有意识地把前面学过的知识点融进新题目里一起练习（比如新知识点的代码练习里要求用到之前学过的函数/操作），不能每个知识点只孤立考自己，避免学一个忘一个。
3. 维护五份固定文档：overview.md / status.md / note.md / rule.md / roadmap.md，不新增其他文件。
4. overview/status/note/rule四份精简撰写，只写核心结论、关键信息、日期节点，不写大段冗余内容；roadmap.md为一次性详细知识节点规划表，不受精简约束，可全面详细，无大方向变更不修改。
5. overview.md：无大方向变更绝不修改。
6. status.md：只记录当前推进事项、待办、注意事项、已完成里程碑，日常更新。
7. note.md：仅在知识点确认掌握后才归档整理，用于后续复习检索。
8. 日常协作只发局部问题，严控上下文Token占用，不整段灌入。
9. 出现规则遗忘、日期搞错、目标跑偏，即判定为幻觉，必须立刻按四份文档内容修正。
10. 会话上下文爆满后弃用当前会话，新会话直接依据四份文档接续，不重新规划。
